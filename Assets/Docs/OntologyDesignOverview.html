Tormia 온톨로지 RPG Core 현황

이 문서는 현재까지 구현된 텍스트 기반 온톨로지 RPG Core를 기획자가 이해하기 쉽게 정리한 개요입니다. 핵심은 “하드코딩 이벤트”가 아니라, 월드의 사실과 규칙이 조합되어 행동, 퀘스트, NPC 반응이 자연스럽게 생성되는 구조입니다.

1. 한 줄 요약

현재 Tormia는 월드에 존재하는 사실을 기반으로, 플레이어가 할 수 있는 행동을 만들고, 행동 결과가 룰을 통해 새 사건, 퀘스트, NPC 감정/관계 변화, 아이템 제작으로 이어지는 텍스트 기반 RPG Core까지 구현되어 있습니다.

2. 현재 플레이 예시

초기 상태
- Player는 FireSword를 장착하고 있음
- Tree/Bush/Reed는 Dry Plant
- Goblin은 Fire를 두려워하고 Village 근처에 있음
- QuestBoard가 존재함

가능 행동
- Attack Tree with FireSword
- Attack Bush with FireSword
- Attack Reed with FireSword
- Talk to Goblin

플레이어가 Tree를 공격
→ Tree state Burning
→ Tree emits Fire
→ Tree emits Smoke
→ Goblin intent Flee
→ Goblin mood Afraid
→ Village state Disturbed
→ QuestBoard offers InvestigateSmokeAndPanic

퀘스트 생성
→ Investigate Smoke and Panic
→ Goal 1: Inspect Tree
→ Goal 2: Help Village

플레이어가 Inspect / Help 완료
→ Player completed_quest InvestigateSmokeAndPanic
→ Village state Recovering
→ Goblin relation to Player becomes Respectful

추가 조합 예시
- Player combined Stone
- Player combined Stick
→ Player has_item StoneAxe
→ combined Stone / combined Stick 제거
→ EventHistory: CraftStoneAxeRule

8. 현재 구조적으로 좋은 점

  • Core는 UnityEngine에 의존하지 않는 순수 C# 구조입니다.
  • Unity scene object는 OntologyObject adapter를 통해 WorldState로 변환됩니다.
  • UI는 OntologyDebugPanel로 분리되어 있습니다.
  • Rule, Quest, ActionCandidate, ActionEffect가 asset database 기반입니다.
  • Relation/Reputation은 문자열 합성이 아니라 별도 entity로 표현됩니다.
  • Action/Event history는 WorldState가 아니라 OntologySession에 분리되어 있습니다.

10. 첫 번째 씨앗 규칙: StoneAxe 제작

이번에 추가한 규칙은 앞으로 자연물, 장비, 신체 변화 규칙을 늘려가기 위한 첫 번째 “씨앗 규칙”입니다. 새 전용 제작 시스템을 만들지 않고, 기존 온톨로지 fact와 RuleDatabase만으로 제작 결과를 추론합니다.

기획 문장

Player가 Stone과 Stick을 조합했다면
두 조합 입력 fact를 소비하고
Player에게 StoneAxe 아이템을 부여한다.

온톨로지 조건과 효과

구분Fact의미
조건Player combined StonePlayer가 Stone을 조합 입력으로 올림
조건Player combined StickPlayer가 Stick을 조합 입력으로 올림
조건not Player has_item StoneAxe이미 가진 아이템을 중복 제작하지 않음
효과Player has_item StoneAxe새 장비 획득
효과remove Player combined Stone조합 입력 Stone 소비
효과remove Player combined Stick조합 입력 Stick 소비

실제 등록된 규칙

Rule id: CraftStoneAxeRule
EventHistory reason: Player -- CRAFTED --> StoneAxe

IF
- Player combined Stone
- Player combined Stick
- Player does not have StoneAxe

THEN
- AddFact Player has_item StoneAxe
- RemoveFact Player combined Stone
- RemoveFact Player combined Stick

검증 결과

ruleCount=10
hasStoneAxe=True
consumedStone=True
consumedStick=True
eventFound=True

[EventHistory] CraftStoneAxeRule: Player -- CRAFTED --> StoneAxe

이 패턴은 앞으로 Wood + Stone → Campfire, Herb + Water → Medicine, Hide + Thread → WarmClothes 같은 조합 규칙으로 그대로 확장할 수 있습니다.

11. 두 번째 씨앗 규칙: Terrain Tile 온톨로지

바닥 타일도 고유한 OntologyId를 가진 entity로 취급합니다. 캐릭터가 어떤 타일 위에 서 있는지만 fact로 기록하면, 타일의 물성에서 캐릭터 상태가 규칙으로 전이됩니다.

기획 의도

if (onSwamp) speed-- 같은 하드코딩 대신
Player standing_on Tile_01
Tile_01 has_element Water
Tile_01 surface Swamp
Tile_01 temperature Freezing
같은 fact 관계로 상태 변화를 추론한다.

대표 타일 fact

Fact의미
Tile_01 has_concept TerrainTileTile_01은 바닥 타일 entity
Player standing_on Tile_01Player가 Tile_01 위에 서 있음
Tile_01 has_element Water타일에 물 속성이 있음
Tile_01 temperature Freezing타일이 영하/빙결 환경임
Tile_01 surface Swamp타일 표면이 늪지임

추가된 전이 규칙

Rule조건결과
TileWaterStatusWetRule?actor standing_on ?tile, ?tile has_element Water?actor status Wet
TileFreezingColdExposureRule?actor standing_on ?tile, ?tile temperature Freezing?actor exposed_to ColdEnvironment
TileSwampMovementSlowRule?actor standing_on ?tile, ?tile surface Swamp?actor movement_state Slowed

검증 목표

Player standing_on Tile_01
Tile_01 has_concept TerrainTile
Tile_01 has_element Water
Tile_01 temperature Freezing
Tile_01 surface Swamp

        ↓ RuleEngine

Player status Wet
Player exposed_to ColdEnvironment
Player movement_state Slowed

이 구조가 4단계 환경 피해 규칙의 징검다리입니다. 3단계에서 수치 연산과 tick 제어가 들어오면, ColdEnvironment 노출 → 매 tick 체력 감소로 이어집니다.

12. 세 번째 씨앗 규칙: Numeric Fact & Tick 생존 루프

수치형 fact를 직접 조정하는 AdjustNumberFact 효과와 tick당 1회 실행을 막는 tick marker 구조를 추가했습니다. 이로써 영하 환경 노출이 실제 체력 감소로 이어지는 생존 루프의 최소 코어가 준비되었습니다.

추가된 엔진 기능

기능역할예시
AdjustNumberFact숫자 object를 가진 fact를 delta만큼 교체Player core_vitality 10 → 9
ChangedFactCountAdd뿐 아니라 Remove, Set, Adjust도 simulation 변경으로 계산수치만 바뀌어도 안정화 판단 가능
tick marker같은 tick에서 반복 피해 방지Player cold_damage_tick Tick_1

ColdExposureVitalityTickRule

IF
- Simulation current_tick ?tick
- ?actor exposed_to ColdEnvironment
- ?actor core_vitality ?vitality
- not ?actor status Warm
- not ?actor cold_damage_tick ?tick

THEN
- AdjustNumberFact ?actor core_vitality -1
- AddFact ?actor cold_damage_tick ?tick

검증 결과

Tile_Cold_01 temperature Freezing
Player standing_on Tile_Cold_01
Player core_vitality 10
Simulation current_tick Tick_1

        ↓ TileFreezingColdExposureRule
        ↓ ColdExposureVitalityTickRule

Player exposed_to ColdEnvironment
Player core_vitality 9
Player cold_damage_tick Tick_1

같은 Tick_1 재실행 → 추가 피해 없음
Tick_2 실행 → Player core_vitality 8
Tick_3 + Player status Warm → 추가 피해 없음

이 구조는 환경 추론과 수치 피해를 분리합니다. 타일 규칙은 “춥다”를 추론하고, tick 규칙은 “이번 tick에 체력이 줄었는가”를 관리합니다.

13. 유저 입장에서의 게임 루프

유저가 체감하는 게임은 “온톨로지”라는 단어를 전면에 드러내지 않습니다. 유저는 월드 안에서 사물을 관찰하고, 가능한 행동을 선택하고, 그 결과로 NPC와 장소가 반응하는 RPG로 경험하게 됩니다.

월드 탐색
→
대상 발견
→
가능 행동 선택
→
월드 반응
→
퀘스트/관계 변화
→
새 행동 해금

기본 플레이 흐름

유저 행동게임 안에서 일어나는 일기획 의미
월드에서 나무, NPC, 퀘스트 보드 등을 본다 각 오브젝트의 성질이 내부적으로 판정된다 “모든 오브젝트는 태그 이상의 의미를 가진다”는 구조
나무를 선택한다 Player가 FireSword를 장착했고 Attack skill이 있으므로 공격 행동이 뜬다 행동 버튼은 고정 UI가 아니라 현재 상황에서 생성된다
Attack Tree with FireSword를 실행한다 나무가 불타고 연기를 내며, 근처 NPC가 겁먹고 마을이 혼란 상태가 된다 하나의 행동이 연쇄 반응을 만든다
퀘스트 보드 또는 상황 변화를 확인한다 Investigate Smoke and Panic 퀘스트가 생성된다 퀘스트는 사전에 강제로 띄우는 것이 아니라 월드 상태에서 발생한다
연기 나는 나무를 조사하고 마을을 돕는다 퀘스트 목표가 완료되고 Village가 Recovering 상태가 된다 퀘스트 목표가 실제 월드 상태와 연결된다
NPC와 다시 상호작용한다 Goblin과 Player의 관계가 Wary/Distrustful/Respectful 등으로 변화한다 NPC 반응은 단발 대사가 아니라 관계 상태로 누적된다
Stone과 Stick을 조합한다 조합 입력 fact가 소비되고 Player가 StoneAxe를 획득한다 제작도 별도 하드코딩 UI가 아니라 월드 fact와 rule로 처리된다
물/늪/빙결 타일 위로 이동한다 타일 속성이 Player 상태로 전이되어 Wet, Slowed, ColdEnvironment 노출이 생긴다 바닥도 독립 entity라서 위치 관계만으로 환경 반응이 발생한다

유저가 느끼게 될 핵심 재미

  • 내 행동이 퀘스트, NPC 감정, 장소 상태에 영향을 준다는 느낌
  • 같은 오브젝트라도 Player의 장비/스킬에 따라 가능한 행동이 달라지는 느낌
  • 퀘스트가 단순 수락형이 아니라 상황에서 발생한다는 느낌
  • NPC가 단순 상점/대화 오브젝트가 아니라 관계를 가진 존재라는 느낌
  • 자연물과 재료가 규칙을 통해 장비와 생존 수단으로 바뀐다는 느낌
  • 내가 밟고 있는 지형 자체가 몸 상태와 생존 조건에 영향을 준다는 느낌
  • 월드가 정해진 컷신이 아니라 규칙으로 반응한다는 느낌

14. 서비스화를 위해 기획자가 해야 할 일

현재는 Core 구조가 준비된 상태입니다. 실제 서비스 가능한 게임으로 만들려면 기획자는 아래 데이터를 채워야 합니다. 중요한 점은 “코드 요청”보다 “온톨로지 데이터와 규칙 설계”가 핵심 업무가 된다는 것입니다.

1. 오브젝트 온톨로지 정의

해야 할 일예시결과
월드 오브젝트 목록 작성Tree, AncientTree, Well, Shrine, Wolf, Merchant, Tile_01게임에 등장할 entity 후보가 정리됨
각 오브젝트의 concept 지정Plant, Dry, Sacred, Creature, NPC, Vendor, TerrainTile룰과 행동 후보가 자동으로 매칭됨
초기 fact 지정Wolf fears Fire, Merchant near Village, Tile_01 surface Swamp시작 시점의 월드 상태가 정해짐

2. Player 성장/능력 설계

  • 초기 장비: 어떤 아이템을 들고 시작하는가
  • 초기 스킬: Attack, Inspect, Help, Talk 외에 무엇이 있는가
  • 해금 구조: 특정 퀘스트 완료 후 어떤 skill/concept/fact가 추가되는가
  • 플레이 스타일: 전투형, 탐사형, 설득형 선택지가 분리되는가

3. Action 설계

유저에게 뜨는 행동은 ActionCandidateDatabase로 정의됩니다.

행동필요 조건행동 결과
AttackAttack skill, 장착 무기, 대상의 성질attacks / attacks_with fact 생성
InspectInspect skill, 조사 가능한 현상inspects fact 생성
TalkTalk skill, 대상이 Creature/NPCtalks_to fact 생성
HelpHelp skill, 대상이 Disturbed 상태helps fact 생성

기획자는 앞으로 “어떤 상황에서 어떤 행동이 보여야 하는가”를 조건 중심으로 정의해야 합니다.

4. Rule 설계

Rule은 월드가 어떻게 반응하는지를 정합니다.

조건:
- ?target has_concept Plant
- ?target has_concept Dry
- ?tool has_concept Fire

결과:
- ?target state Burning
- ?target emits Fire

기획자는 아래 유형의 규칙을 설계해야 합니다.

  • 환경 반응: 불, 물, 독, 빛, 소리, 날씨
  • 지형 반응: 물 타일, 늪 타일, 빙결 타일, 독성 타일, 성역 타일
  • NPC 감정: 공포, 신뢰, 분노, 호기심
  • 장소 상태: 안전, 혼란, 점령, 회복, 봉쇄
  • 아이템 상호작용: 태움, 수리, 개방, 강화, 오염, 조합 제작
  • 생존/신체 변화: 추위 피해, 온기, 독, 회복, 피로
  • 퀘스트 hook: 어떤 상황이 퀘스트를 발생시키는가

5. Quest 설계

Quest는 고정 대본이 아니라 월드 상태에서 발생합니다.

항목기획자가 정할 내용
발생 조건어떤 fact가 생기면 퀘스트가 뜨는가
목표어떤 행동/월드 상태가 목표 완료로 인정되는가
완료 결과완료 후 장소, NPC, 아이템, 평판이 어떻게 변하는가
중복/분기같은 퀘스트가 여러 장소에서 발생할 때 하나로 묶을지 분리할지

6. NPC 관계/감정 설계

  • NPC별 기본 성향: Cowardly, Brave, Greedy, Loyal 등
  • 감정 상태: Calm, Afraid, Angry, Curious, Grateful
  • 관계 상태: Wary, Distrustful, Neutral, Respectful, Friendly
  • 관계 변화 조건: 대화, 도움, 공격, 배신, 선물, 퀘스트 완료
  • 관계별 대사/행동 차이

7. 콘텐츠 스프레드시트 필요

서비스 단계로 가려면 아래 표 형태의 데이터 관리가 필요합니다.

8. 밸런싱/QA 기준

  • 행동 후보가 너무 많이 뜨지 않는가
  • 퀘스트가 의도치 않게 중복 발생하지 않는가
  • NPC 관계가 모순 상태로 쌓이지 않는가
  • 상태 교체가 자연스러운가: Disturbed → Recovering → Safe 등
  • 유저가 원인과 결과를 이해할 수 있는가
  • 한 행동이 너무 많은 연쇄 반응을 만들어 혼란스럽지 않은가

17. Ontology Map Editor: 기획 데이터 중심 맵 툴

기존 Terrain Map Tool이 씬 GameObject 생성기라면, 새 Ontology Map Editor는 기획자가 타일 템플릿과 맵 배치를 데이터 asset으로 설계하고 검증하는 전용 툴입니다. 런타임 OntologyFact를 직접 직렬화하지 않고, string 기반 predicate/object 템플릿을 저장한 뒤 검증 시 Tile_x_y subject로 결합합니다.

TilePlacementData좌표, templateId, uniqueFacts를 저장하는 한 칸 배치 정보

18. Player 위치 기반 standing_on 자동 갱신

인게임 플레이어 Transform 위치를 ontology tile 좌표로 변환해 Player standing_on Tile_x_y fact를 자동 갱신하는 런타임 tracker를 추가했습니다. 이 컴포넌트가 타일 이동을 감지하면 이전 standing_on fact를 제거하고 새 fact를 넣은 뒤, 선택적으로 RuleEngine을 즉시 실행합니다.

추가된 컴포넌트

Assets/Scripts/Ontology/Unity/OntologyPlayerPositionTracker.cs

Inspector 설정

필드설명
BootstrapOntologyWorldBootstrap 직접 참조. 비어 있으면 scene에서 자동 탐색
Player추적할 플레이어 Transform. 비어 있으면 자기 Transform 사용
Grid Origin격자 원점 Transform. 비어 있으면 world zero 기준
Tile Size한 타일의 월드 크기. 위치 좌표를 나눠 grid x/y 계산
Actor Id기본값 Player. 생성되는 fact의 subject
Tile Id Prefix기본값 Tile. 결과 subject는 Tile_x_y
Update Interval위치 확인 주기
Run Simulation On Tile Changed타일 이동 직후 RuleEngine 실행 여부

좌표 변환 방식

local = player.position - gridOrigin.position
x = Floor(local.x / tileSize)
y = Floor(local.z / tileSize)

Unity 3D 공간에서는 world Z축을 ontology grid Y축으로 사용합니다.

예:
player position = (2.4, 0, 3.1)
tileSize = 1
→ x=2, y=3
→ Tile_2_3

팩트 교체와 이벤트

이전 위치:

새 위치:
Player standing_on Tile_2_3

처리:
- RemoveFact Player standing_on Tile_2_2
- AddFact Player standing_on Tile_2_3
- EventHistory 추가

EventType:
PlayerStandingOnChanged

Reason:
Player moved from Tile_2_2 to Tile_2_3

→ 위치를 (x * tileSize, 0, y * tileSize)로 배치
→ OntologyObject 자동 부착
→ concepts = TerrainTile
→ facts = grid_x, grid_y, tile_type, baseFacts, uniqueFacts
→ OntologyWorldBootstrap.ResetWorld(false)
→ WorldState에 Tile_* facts 주입
